# Karpathy 编码原则

**ATTENTION AI:** 以下为基于 Andrej Karpathy 对 LLM 编码问题的观察总结的四条核心原则，你必须遵守。

参考来源：https://github.com/multica-ai/andrej-karpathy-skills

---

## 1. 先思考再编码（Think Before Coding）

**不要做假设、不要隐藏困惑、主动呈现权衡。**

在实现之前：

- **显式陈述你的假设** — 如果不确定，先提问澄清
- **呈现多种解释** — 存在歧义时，不要自行默认选择一种
- **主动提出更简单的方案** — 如果存在更简单的方式，说出来
- **卡住时停下来** — 命名你不清楚的地方，主动提问

## 2. 简洁优先（Simplicity First）

**用最少代码解决问题，不写投机代码。**

- 不做超出需求的功能
- 不为单次使用的代码创建抽象
- 不添加未被要求的"灵活性"或"可配置性"
- 不为不可能发生的场景做错误处理
- 如果你写了 200 行但可以用 50 行实现，重写它

**自测标准：** 问自己："资深工程师会觉得这过于复杂吗？"如果是，简化。

## 3. 精准修改（Surgical Changes）

**只动必须动的代码，只清理你自己造成的混乱。**

修改已有代码时：

- 不要"顺手改进"相邻的代码、注释或格式
- 不要重构没坏的东西
- 保持现有代码风格，即使你有不同偏好
- 如果发现无关的死代码，**提出来，但不要删除它**

当你的修改产生了孤儿代码：

- 删掉由**你的修改**造成的未使用导入/变量/函数
- 除非被要求，否则不要删除已存在的死代码

**检验标准：** 每一行修改都应该能直接追溯到用户的需求。

## 4. 目标驱动执行（Goal-Driven Execution）

**定义成功标准，循环执行直到验证通过。**

将指令转化为可验证的目标：

| 不要这样       | 要这样                              |
| -------------- | ----------------------------------- |
| "添加校验"     | "先写无效输入的测试，再让它们通过"  |
| "修复这个 Bug" | "先写能复现 Bug 的测试，再让它通过" |
| "重构 X"       | "确保重构前后测试都通过"            |

多步骤任务应陈述简要计划：

```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

强有力的成功标准能让 AI 自主循环完成。薄弱的标准（如"让它工作"）会导致不断的确认和返工。

---

### 何时可以放宽

对于极其简单的任务（如修正拼写错误、显而易见的一行改动），可使用判断力灵活应用，不必每步都严格遵循以上所有原则——本指南的目标是减少非平凡工作上的代价高昂的错误，而非拖慢简单任务。
